# ML-to-Frontend Integration Reference

This document is the canonical integration spec for how ML outputs are exposed
to the frontend.

It covers:
- exact data sources
- how each frontend API endpoint is built
- which frontend components consume each endpoint
- how RUL and replacement date are calculated
- how to refresh the data end-to-end
- how to extend the integration for new models (for example scheduler outputs)

## Integration Architecture

There are two integration patterns in use.

1. Live computed API from ML artifacts (no prebuilt JSON per request)
- Used by: `/api/v1/planes`
- Source: latent SOH CSV tables in `ml_workspace/latent_soh/output/plane_<id>/latent_soh_event_table.csv`
- Purpose: fast fleet cards with latest SOH/trend/flight/charge counters

2. Batch snapshot API (JSON produced by exporter, then served)
- Used by: `/api/v1/planes/[planeId]/health`, `/predictions`, `/soh-trend`, `/recommendations`, `/flights`, `/glossary`, `/learn/baseline`
- Source JSONs: `frontend/public/snapshots/*.json`
- Generated by:
  - `ml_workspace/integration/export_snapshots.py` (for KPI/trend/history)
  - pre-existing snapshot JSONs for recommendations/flights/glossary/learn

## Core Data Sources

### Latent SOH source of truth
- Path pattern:
  - `ml_workspace/latent_soh/output/plane_166/latent_soh_event_table.csv`
  - `ml_workspace/latent_soh/output/plane_192/latent_soh_event_table.csv`
- Key columns consumed:
  - `event_datetime`
  - `event_type`
  - `latent_soh_filter_pct` (causal latent SOH)
  - `latent_soh_smooth_pct` (for history display where available)
  - `soc_max_pct`, `soc_mean_pct`, `soc_min_pct`
  - telemetry columns (`voltage_mean_v`, `current_abs_mean_a`, `avg_cell_temp_mean_c`)

### Forecast model outputs for RUL integration
- Root:
  - `ml_workspace/soh_forecast/output/multihorizon_runner_plane_166/`
- Required artifacts:
  - `best_models_by_horizon.json`
  - `latent_flight_<H>/latent_flight_<H>_predictions.csv` for each horizon
- Horizons currently used:
  - `1,5,10,15,20` (configurable via exporter arg)

### Frontend snapshot directory
- `frontend/public/snapshots/`
- Files generated/consumed:
  - `plane_<id>_kpis.json`
  - `plane_<id>_soh_trend.json`
  - `plane_<id>_soh_history.json`
  - `plane_<id>_flights.json`
  - `plane_<id>_recommendations_<YYYY_MM>.json`
  - `planes.json`
  - `glossary.json`
  - `learn_baseline_plane_<id>.json`

## End-to-End Data Flow

1. Build/refresh latent SOH output tables.
2. Train multihorizon forecast models and benchmark/select best models per horizon.
3. Run `export_snapshots.py` to create/update `plane_<id>_kpis.json`, trend, history.
4. Frontend API routes load these snapshots (or live latent CSV for `/planes`).
5. React components fetch API routes via `frontend/lib/adapters/api-client.ts`.

## API Endpoint Data Lineage

### `/api/v1/planes`
- Route: `frontend/app/api/v1/planes/route.ts`
- Builder: `frontend/lib/live-plane-summaries.ts`
- Source:
  - live latent CSV per plane for SOH/trend/counters
  - `frontend/public/snapshots/planes.json` for plane metadata (`registration`, `aircraftType`, list of plane IDs)
- Fallback:
  - if latent CSV missing, pulls `sohCurrent/sohTrend30` from `plane_<id>_kpis.json`
- Used by:
  - `frontend/components/plane/plane-index.tsx`
  - `frontend/components/learn/learn-simulator.tsx`
  - `frontend/components/mission-game/mission-game-shell.tsx`

### `/api/v1/planes/[planeId]/health`
- Route: `frontend/app/api/v1/planes/[planeId]/health/route.ts`
- Source:
  - `frontend/public/snapshots/plane_<id>_kpis.json` -> `health`
- Normalization/enrichment:
  - fills null route fields with safe defaults
  - computes fallback `timeSinceLastFlightHours` if absent
  - remaps risk bands and health labels from SOH via `frontend/lib/soh-health-bands.ts`
- Used by:
  - `frontend/components/plane/plane-dashboard.tsx`

### `/api/v1/planes/[planeId]/predictions`
- Route: `frontend/app/api/v1/planes/[planeId]/predictions/route.ts`
- Source:
  - `frontend/public/snapshots/plane_<id>_kpis.json` -> `prediction`
- Used by:
  - `frontend/components/plane/plane-dashboard.tsx`

### `/api/v1/planes/[planeId]/soh-trend?window=...`
- Route: `frontend/app/api/v1/planes/[planeId]/soh-trend/route.ts`
- Source:
  - primary: `plane_<id>_soh_history.json`
  - fallback: `plane_<id>_soh_trend.json`
- Transformation:
  - window filter (`30d`, `90d`, `1y`)
- Used by:
  - `frontend/components/plane/plane-dashboard.tsx` (SOH history chart)

### `/api/v1/planes/[planeId]/recommendations`
- Route: `frontend/app/api/v1/planes/[planeId]/recommendations/route.ts`
- Source:
  - month-specific recommendations snapshot
- Used by:
  - `frontend/components/plane/plane-dashboard.tsx`

## Exporter Outputs and Field Derivation

Exporter module:
- `ml_workspace/integration/export_snapshots.py`

Generated outputs:
- `plane_<id>_kpis.json`
- `plane_<id>_soh_trend.json`
- `plane_<id>_soh_history.json`

### `plane_<id>_kpis.json -> health`

Derived from latest latent table rows:
- `sohCurrent`: latest `latent_soh_filter_pct` (causal)
- `sohTrend30` / `sohTrend90`: window deltas on `latent_soh_filter_pct`
- `riskBand` / `healthLabel`:
  - `80-100 => low/healthy`
  - `40-80 => medium/medium`
  - `20-40 => watch/watch`
  - `0-20 => critical/critical`
- `currentChargeSoc`:
  - SOC from most recent charge event (`soc_max_pct`, fallback `soc_mean_pct`, then `soc_min_pct`)
  - fallback to latest row SOC if no charge event
- `pack.soc`: same as `currentChargeSoc`
- `pack.voltage/current/tempAvg`: latest telemetry summaries
- `lastFlight`: last `event_type == flight` row projected to API contract

### `plane_<id>_kpis.json -> prediction`

RUL and replacement are model-driven:
- `rulCyclesPred`:
  - load best model per horizon from `best_models_by_horizon.json`
  - for each selected horizon:
    - load `latent_flight_<H>_predictions.csv`
    - use best-model prediction column named by model id
    - compute per-row SOH delta: `pred_next_soh - latent_soh_filter_pct`
    - convert to per-flight by dividing by horizon `H`
    - use recent rows and robust median of negative deltas
  - combine horizon estimates with inverse-`delta_mae` weighting
  - fallback to empirical recent flight deltas if model estimate unavailable
  - final formula:
    - `rulCyclesPred = max(0, (current_soh - eol_soh) / abs(soh_drop_per_flight))`
- `rulDaysPred`:
  - estimate flights/day from recent flight cadence window
  - `rulDaysPred = ceil(rulCyclesPred / flights_per_day)`
- `replacementDatePred`:
  - `today_utc + rulDaysPred`
- `confidence`:
  - blend of model estimate confidence and cadence confidence

Default EOL:
- `--eol-soh 0` (adjusted SOH scale used in frontend)

### Trend/history files

- `plane_<id>_soh_trend.json`:
  - daily median points from `latent_soh_filter_pct`
- `plane_<id>_soh_history.json`:
  - daily median points from `latent_soh_smooth_pct` when available
  - fallback to `latent_soh_filter_pct`

## Frontend Consumption Map

### `/planes` page (`frontend/components/plane/plane-index.tsx`)
Uses `/api/v1/planes` values:
- SOH
- 30d trend
- flights count
- charging events count
- risk badge

### `/planes/[id]` dashboard (`frontend/components/plane/plane-dashboard.tsx`)
Uses:
- `/health` for current SOH, charge, pack metrics, label, last flight
- `/predictions` for RUL cycles/days and replacement date
- `/soh-trend` for history chart
- `/recommendations` for monthly calendar/cards
- `/weather` and `/charging-cost` for contextual operating guidance

## Command Runbook

### 1) Train/refresh forecast outputs

```bash
.venv/bin/python -m ml_workspace.soh_forecast.train_horizon_models \
  --primary-plane 166 \
  --holdout-plane 192 \
  --horizons 1,5,10,15,20 \
  --device cpu
```

### 2) Export integration snapshots

```bash
.venv/bin/python -m ml_workspace.integration.export_snapshots \
  --plane-ids 166,192 \
  --forecast-root ml_workspace/soh_forecast/output/multihorizon_runner_plane_166 \
  --rul-horizons 1,5,10,15,20 \
  --eol-soh 0
```

### 3) Start frontend

```bash
cd frontend
npm run dev
```

### 4) Sanity check APIs

```bash
curl -s http://localhost:3000/api/v1/planes | python3 -m json.tool
curl -s http://localhost:3000/api/v1/planes/166/health | python3 -m json.tool
curl -s http://localhost:3000/api/v1/planes/166/predictions | python3 -m json.tool
```

## Leakage and Causality Policy

For forecast-driving values in frontend:
- use causal latent SOH (`latent_soh_filter_pct`)
- do not use RTS-smoothed values for forecasting/RUL computation

RTS values may still be used for visualization history where intended.

## Known Limitations

- Snapshot data is batch refreshed, not auto-regenerated on every frontend request.
- Recommendations and flights endpoints are still snapshot-backed.
- Weather and charging cost endpoints are partially live and may fallback to modeled values.
- Mission-game scoring includes heuristic components; it is not yet a direct optimizer integration.

## Extension Pattern for New Models

To add a new backend model integration (for example schedule optimizer):

1. Write model output artifact under `ml_workspace/<new_module>/output/`.
2. Extend `ml_workspace/integration/export_snapshots.py` to transform it into frontend contract JSON.
3. Add/extend snapshot files in `frontend/public/snapshots/`.
4. Add/extend API route under `frontend/app/api/v1/...`.
5. Add/extend adapter function in `frontend/lib/adapters/api-client.ts`.
6. Consume from the React component with `useQuery`/`useMutation`.
7. Document source + transformation + UI usage in this README.
